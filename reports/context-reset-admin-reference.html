1b. Integration: The Auto-Continue Loop

context-reset is not a Stop hook. It's a tool that Claude calls when it decides context is full. The auto-continue loop is what drives autonomous operation:

┌─ Claude Code Session ──────────────────────────────────────┐ │ │ │ 1. Claude works on TODO.md tasks │ │ 2. Claude tries to stop │ │ 3. Stop hook (auto-continue.js): │ │ → reads stop-message.txt │ │ → returns {decision: "block", reason: message} │ │ 4. Claude sees: "DO NOT STOP. Do the next task." │ │ 5. Claude continues working │ │ 6. Context fills up │ │ 7. Claude saves state to TODO.md │ │ 8. Claude runs: context-reset --project-dir $DIR │ │ → saves SESSION_STATE.md (conversation summary) │ │ → opens fresh tab with new Claude process │ │ → verifies new session is working │ │ → kills old tab │ │ 9. New session reads SESSION_STATE.md + TODO.md │ │ 10. Back to step 1 (infinite loop) │ └────────────────────────────────────────────────────────────┘

Recommended: hook-runner

hook-runner is a modular hook system that manages all Claude Code hooks as .js files in folders. The auto-continue.js module ships with hook-runner's starter workflow (48 modules including git safety, secret scanning, commit quality).

npx grobomo/hook-runner --yes        # install with starter workflow
pip install git+https://github.com/grobomo/context-reset

Key files

FilePurpose
modules/Stop/auto-continue.jsBlocks every stop, reads message from stop-message.txt, returns {decision: "block"}
modules/Stop/stop-message.txtUser-authored prompt — tells Claude what to do instead of stopping. Separate file so prompt iteration doesn't touch code.
run-stop.jsRunner that loads all modules/Stop/*.js modules
run-hidden.jsWindows wrapper — runs hooks with hidden console window to prevent focus steal

Why separate the message file?

The stop message is a user-authored prompt artifact, iterated over 15+ versions. Keeping it in a separate .txt file means: (1) prompt iteration doesn't risk breaking JavaScript, (2) Claude can't accidentally rewrite it during code changes, (3) it's easy to customize per-team or per-project.

Standalone alternative

Users who prefer not to install hook-runner can create a minimal auto-continue.js that reads a separate message file and returns {decision: "block"}. See the README for the complete standalone setup.

Common mistake: Do NOT wire context-reset as a Stop hook directly. Stop hooks fire on every stop (not just context-full), context-reset outputs no {decision: "block"} JSON, and it blocks for 15-45s during verification. The correct pattern is: auto-continue blocks stops → Claude works → Claude calls context-reset when needed.

2. Key Internals

2.1 Platform Detection

FlagDetection MethodValue
IS_WINsys.platform == "win32"True on Windows (native)
IS_MACsys.platform == "darwin"True on macOS
IS_WSL/proc/version contains "microsoft"True inside WSL only

Evaluation order in build_launch_cmd: IS_WIN → IS_WSL → IS_MAC → else (Linux).

2.2 Prompt-File Approach

All platforms write the prompt to {project_dir}/.claude-next-prompt. The new session's shell reads it, deletes it, and passes it to claude. This avoids:

  • WT ; command separator splitting prompts (Windows)
  • Shell quote escaping in nested invocations (all platforms)
  • Argument length limits (SSH sessions)

2.4 Session State Extraction

_tail_lines() reads the last N lines from JSONL using efficient reverse-read (no full file load). _parse_and_render_tail() converts to readable conversation text. Smart truncation keeps first ~25% + last ~75%, drops the middle. Capped at ~8K tokens.

10. Security & Safety

ControlDescription
Two-phase verificationWon't kill old tab until new Claude is confirmed working (process count + transcript activity)
Multi-Claude guardRefuses to kill a shell that owns >1 Claude processes
Duplicate session guardPre-launch check: if target project transcript was written <10s ago, refuses to spawn (PR #64)
OS-level focus restoreBackground monitor thread polls for 5s using Win32 GetForegroundWindow/SetForegroundWindow, catches 1-3 async WT focus steals per launch (PRs #63, #66)
Kill retrytaskkill captures stderr, retries once on failure, verifies PID death via os.kill(pid, 0) (PR #63)
Process tree walkFinds the correct terminal shell PID, not inner Bash tool shells
Detached kill (Windows)taskkill /T runs in a detached subprocess so it doesn't kill the caller
Audit logging~/.claude/context-reset/YYYY-MM-DD.log with 7-day retention
Prompt-file cleanup.claude-next-prompt deleted by the new session immediately after reading
Pre-trust safetyOnly writes to ~/.claude.json if no ancestor is already trusted
No secrets in repoCI secret-scan on every push (SSH keys, tokens, PII patterns)
